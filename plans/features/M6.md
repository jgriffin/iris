## Thesis

Iris has proven the built-in Vision surface (M5). M6 extends `Detector` past
Vision to **arbitrary Core ML models** ‚Äî the YOLO family first ‚Äî so downstream
apps can run their own detector through the same `DetectorCatalog` / overlay /
inspector / tuning machinery they already get for free. Ties to
the [`BOARD.md`](../BOARD.md) ¬ßMilestones M6 entry ("Custom models") + [`BRIEF.md`](../BRIEF.md)
¬ß3 `IrisDetection`'s Core ML bullet ("generic `MLModel` wrapper with
pluggable output decoders"): a Core ML adapter with a YOLO-style output decoder,
a model-swap affordance, and ‚Äî once planned as a stretch ‚Äî Foundation Models
captioning ([`BRIEF.md`](../BRIEF.md) ¬ß3 `Captioner`), now **dropped** (see ¬ßP4:
Foundation Models is text-only, so on-device captioning needs a VLM ‚Äî off-plan).

Iris runs models through Core ML / Vision; it does **not** run PyTorch. Every
external model is converted to an `.mlpackage` offline by Python tooling that
lives *outside* the Swift package and never becomes a SwiftPM dependency. The
conversion pipeline is P1 (done); the Swift detector is P2‚ÄìP3.

## Scope (this milestone)

- **Conversion pipeline** (P1, ‚úÖ): reusable `tools/model-export/` tooling +
  runbook, verified YOLO conversion paths, converted artifacts under
  `data/models/`.
- **`CoreMLDetector` + `VisionObjectDecoder`** (P2): one Core ML detector with a
  pluggable `OutputDecoder` seam, proven on **YOLOv12** (zero-Swift-decode path)
  end-to-end in a demo.
- **`YOLOEnd2EndDecoder` + model-swap loading** (P3): the second decoder
  (YOLO26 raw tensor) through the same seam, plus catalog/loading for bundled +
  file-picked models.
- **Captioning** (P4, üö´ **dropped 2026-05-26**): was `Captioner` protocol + a
  Foundation Models backend. Dropped ‚Äî Foundation Models is text-only (no image
  input) and Vision has no caption request, so an on-device image captioner needs
  a VLM (off-plan); the user deprioritized captioning. See ¬ßP4.

Off the critical path (already in [`QUESTIONS.md`](../QUESTIONS.md), not
duplicated here): the **RF-DETR Core ML spike** (no native export; direct
PyTorch‚ÜíCore ML patched-fork route, FP32-only, needs a `DETRSetPredictionDecoder`)
and other YOLO sizes.

## Design spine ‚Äî one detector, a pluggable `OutputDecoder` seam

The decisive fork is **how the converted model's output is decoded**, and it's a
fact about the exported artifact, not a Swift choice (verify with
`inspect_model.py` before writing any Swift). Apple Vision auto-decodes a Core ML
model into ready-made observations **only** when the model is a pipeline ending
in a `NonMaximumSuppression` stage with `coordinates` + `confidence` outputs
("**path A**"); anything else comes back as a raw `MLMultiArray` Iris decodes in
Swift ("**path B**").

Model the two paths as **one detector with a swappable decoder**, not two
detectors (per [RECOMMENDATIONS ¬ß2](../../explorations/2026-05-25-coreml-model-conversion/RECOMMENDATIONS.md)):

```swift
// Sketch ‚Äî not final. Lives in Sources/Iris/Detection/CoreML/.
public protocol OutputDecoder: Sendable {
    func decode(_ observations: [any VisionObservation],
                frameSize: CGSize,
                modelID: String) throws -> [Detection]
}

public struct CoreMLDetector<Decoder: OutputDecoder>: Detector {
    let model: MLModel            // compiled .mlmodelc, loaded with MLModelConfiguration.computeUnits
    let inputSize: CGSize         // model-fixed (YOLO 640¬≤; RF-DETR per-variant) ‚Äî pins crop/scale
    let cropAndScale: ...         // aspect-preserving scale-to-fit (letterbox) ‚Äî NEVER hardcode 640
    let decoder: Decoder
    public var availability: DetectorAvailability   // .modelNotReady until the .mlmodelc is present
    public let modelIdentifier: String
}
```

Three decoders, additive through the one seam:

- **`VisionObjectDecoder`** ‚Äî path A. Reads `boundingBox` + best label +
  `confidence` off each recognized-object observation. One implementation reused
  by *every* `nms=True` Detect export, any size, any class set. The default and
  the only decoder P2 ships.
- **`YOLOEnd2EndDecoder`** ‚Äî path B, YOLO26 `[1,300,6]`. **Trivial: threshold +
  scale the ‚â§300 rows `[x1, y1, x2, y2, conf, classIndex]` (xyxy in 640-px space,
  NOT xywh, NOT normalized), NO NMS** ‚Äî the one-to-one head self-dedupes. The
  converted export carries **no embedded labels** (empty metadata), so labels are
  supplied **externally** via `YOLOEnd2EndDecoder(labels:)` + the canonical COCO-80
  constant ‚Äî *not* read from `userDefined` `names`. Shipped in P3.
- **`DETRSetPredictionDecoder`** ‚Äî path B, RF-DETR (later, additive; off the
  critical path). sigmoid ‚Üí flatten top-k ‚Üí cxcywh‚Üíxyxy, no NMS.

**Pin each model's fixed input size + aspect-preserving scale-to-fit** in the
catalog/detector config ‚Äî never hardcode 640 (RF-DETR variants are 384‚Äì880). A
crop/scale that doesn't match how the model was trained silently corrupts boxes.

## What's reused vs. new

**Reused (already exists, M2‚ÄìM5):** `Detector` / `TunableDetector`,
`DetectorCapabilities`, `DetectorCatalog` / `DetectorCatalogEntry` /
`ActiveDetectorSession` (the M5¬∑P4 selection layer ‚Äî Core ML models register as
entries exactly like the built-in Vision ones), `DetectorPipeline`, the overlay
(`DetectionLayer` + `VideoGeometry`), the `DetectionInspector`, and the
`ResultStore` / `DetectionCache` path.

**New (this milestone):** `tools/model-export/` Python tooling (P1, done);
`Sources/Iris/Detection/CoreML/` ‚Äî `CoreMLDetector` + `OutputDecoder` protocol +
`VisionObjectDecoder` (P2) + `YOLOEnd2EndDecoder` (P3); catalog entries + model
loading (bundle + file-picker) (P3). ~~`Captioner` + Foundation Models backend (P4)~~
‚Äî P4 dropped (see ¬ßP4).

### P3 ‚Äî `YOLOEnd2EndDecoder` (Path B) + model-swap catalog/loading  ‚úÖ

The second decoder through the same seam + the loading story:

- **`YOLOEnd2EndDecoder`**  ‚úÖ ‚Äî decodes YOLO26's raw `[1,300,6]`: threshold + scale

### P4 ‚Äî Captioning  üö´ dropped

_Dropped 2026-05-26._ Foundation Models is **text-only** ‚Äî `LanguageModelSession.respond(to:)`
takes a `Prompt`/`String` and returns `Response<String>` (or a structured `Generable`);
`Prompt` is built purely from text, there is **no `CGImage`/`CVPixelBuffer`/`CIImage`
image-input path** (verified against the MacOSX26.5 SDK). Vision likewise has **no
natural-language caption request** (its catalog is detection/classification/saliency/
text-recognition; the closest, `ClassifyImageRequest`, returns labels, not a caption).
So an on-device image‚Üítext captioner needs a **vision-language model (VLM) converted to
Core ML** ‚Äî a separate, off-plan lift, not a "Foundation Models backend" (which corrects
the original assumption here and in [`BRIEF.md`](../BRIEF.md) ¬ß3 `Captioner`). Combined
with the user deprioritizing captioning, P4 is dropped (not deferred-as-stretch). A future
captioning milestone would start from a Core ML VLM. The original plan ‚Äî a `Captioner`
protocol (`image ‚Üí text`, separate from `Detector` per the 2026-05-20 decision in
[`DECISIONS.md`](../DECISIONS.md)) ‚Äî still stands as the *shape* such a milestone would take.

## Open design questions

- üìå **Runtime-tunable Core ML thresholds ‚Äî resolved for P2, implemented in P3.**
  Path A (`nms=True`) **bakes the IoU/confidence thresholds at export time**, so
  runtime-tunable thresholds would force Path B (raw tensor + Swift NMS/threshold)
  or a re-export per threshold set ‚Äî which changes *which decoder* the YOLO catalog
  entries use. **P2 shipped the baked-threshold Path-A `VisionObjectDecoder` and
  conforms `CoreMLDetector` to `Detector` only (no runtime knobs).** **P3 then
  added the tunable variant on the path-B side:** a `TunableOutputDecoder:
  OutputDecoder` sub-protocol + `extension CoreMLDetector: TunableDetector where
  Decoder: TunableOutputDecoder`, so `YOLOEnd2EndDecoder` exposes a runtime
  confidence-threshold knob while path A stays non-tunable (the conditional
  conformance is the seam). `apply` tiers: no-op ‚Üí `.view`; raise threshold ‚Üí
  `.filter` (higher-conf rows are a strict subset of the cache ‚Äî a pure post-hoc
  predicate, no re-inference); lower ‚Üí `.detector(rebuilt:)` (previously-dropped
  rows aren't cached ‚Äî rebuilds the detector around the SAME compiled container, no
  model recompile). See [`DECISIONS.md`](../DECISIONS.md) (2026-05-26) and
  [`QUESTIONS.md`](../QUESTIONS.md) `[answered 2026-05-26]`.
- üìå **iOS/macOS 26 auto-decode observation type ‚Äî resolved.** Confirmed at runtime
  in P2 against the MacOSX26.5 SDK: `CoreMLRequest.Result == [any VisionObservation]`,
  and an NMS object-detection pipeline yields concrete **`RecognizedObjectObservation`**
  (logged via a temporary probe) ‚Äî `boundingBox: NormalizedRect` (normalized
  lower-left, matches `Detection.boundingBox` ‚Äî no flip), best-first `labels`,
  `confidence`. The auto-decode *rule* (model output spec, not request type) held.

## Risks

- **Box misalignment from crop/scale mismatch.** The classic Vision‚ÜîYOLO trap ‚Äî
  a center-crop/stretch that doesn't match the model's letterbox training corrupts
  boxes. Mitigation: pin aspect-preserving scale-to-fit per model; the converted
  model declares a fixed-size image input with internal `/255`.
- **Path assumption vs. artifact reality.** Never assume the decode path ‚Äî the
  RUNBOOK's `inspect_model.py` step makes it a checked fact (`coordinates` +
  `confidence` pipeline ‚áí A; a `multiArrayType` ‚áí B). A doc-only pass already got
  YOLO26's path wrong once; empirical verification corrected it.
- **Foundation Models scope (P4).** ~~Captioning is a stretch; don't let it block the
  Core ML detector close.~~ Moot ‚Äî P4 dropped (Foundation Models is text-only; see ¬ßP4).
  The Core ML detector (P1‚ÄìP3) closed M6 on its own.
