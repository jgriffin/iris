# M6 — Custom models + captioning

<!-- Working plan. Lifetime ~ this milestone; LOG.md keeps the trail. Status vocab per WORKFLOW.md §"Status trees". -->
_Defined · 2026-05-25_

## Thesis

Iris has proven the built-in Vision surface (M5). M6 extends `Detector` past
Vision to **arbitrary Core ML models** — the YOLO family first — so downstream
apps can run their own detector through the same `DetectorCatalog` / overlay /
inspector / tuning machinery they already get for free. Ties to
[`BRIEF.md`](../BRIEF.md) "Custom models + captioning" (the M6 milestone-path
entry + §3 `IrisDetection`'s Core ML bullet: "generic `MLModel` wrapper with
pluggable output decoders"): a Core ML adapter with a YOLO-style output decoder,
a model-swap affordance, and — as a stretch — Foundation Models captioning
([`BRIEF.md`](../BRIEF.md) §3 `Captioner`).

Iris runs models through Core ML / Vision; it does **not** run PyTorch. Every
external model is converted to an `.mlpackage` offline by Python tooling that
lives *outside* the Swift package and never becomes a SwiftPM dependency. The
conversion pipeline is P1 (done); the Swift detector is P2–P3.

## Scope (this milestone)

- **Conversion pipeline** (P1, ✅): reusable `tools/model-export/` tooling +
  runbook, verified YOLO conversion paths, converted artifacts under
  `data/models/`.
- **`CoreMLDetector` + `VisionObjectDecoder`** (P2): one Core ML detector with a
  pluggable `OutputDecoder` seam, proven on **YOLOv12** (zero-Swift-decode path)
  end-to-end in a demo.
- **`YOLOEnd2EndDecoder` + model-swap loading** (P3): the second decoder
  (YOLO26 raw tensor) through the same seam, plus catalog/loading for bundled +
  file-picked models.
- **Captioning** (P4, stretch): `Captioner` protocol + a Foundation Models
  backend.

Off the critical path (already in [`QUESTIONS.md`](../QUESTIONS.md), not
duplicated here): the **RF-DETR Core ML spike** (no native export; direct
PyTorch→Core ML patched-fork route, FP32-only, needs a `DETRSetPredictionDecoder`)
and other YOLO sizes.

## Design spine — one detector, a pluggable `OutputDecoder` seam

The decisive fork is **how the converted model's output is decoded**, and it's a
fact about the exported artifact, not a Swift choice (verify with
`inspect_model.py` before writing any Swift). Apple Vision auto-decodes a Core ML
model into ready-made observations **only** when the model is a pipeline ending
in a `NonMaximumSuppression` stage with `coordinates` + `confidence` outputs
("**path A**"); anything else comes back as a raw `MLMultiArray` Iris decodes in
Swift ("**path B**").

Model the two paths as **one detector with a swappable decoder**, not two
detectors (per [RECOMMENDATIONS §2](../../explorations/2026-05-25-coreml-model-conversion/RECOMMENDATIONS.md)):

```swift
// Sketch — not final. Lives in Sources/Iris/Detection/CoreML/.
public protocol OutputDecoder: Sendable {
    func decode(_ observations: [any VisionObservation],
                frameSize: CGSize,
                modelID: String) throws -> [Detection]
}

public struct CoreMLDetector<Decoder: OutputDecoder>: Detector {
    let model: MLModel            // compiled .mlmodelc, loaded with MLModelConfiguration.computeUnits
    let inputSize: CGSize         // model-fixed (YOLO 640²; RF-DETR per-variant) — pins crop/scale
    let cropAndScale: ...         // aspect-preserving scale-to-fit (letterbox) — NEVER hardcode 640
    let decoder: Decoder
    public var availability: DetectorAvailability   // .modelNotReady until the .mlmodelc is present
    public let modelIdentifier: String
}
```

Three decoders, additive through the one seam:

- **`VisionObjectDecoder`** — path A. Reads `boundingBox` + best label +
  `confidence` off each recognized-object observation. One implementation reused
  by *every* `nms=True` Detect export, any size, any class set. The default and
  the only decoder P2 ships.
- **`YOLOEnd2EndDecoder`** — path B, YOLO26 `[1,300,6]`. **Trivial: threshold +
  scale the ≤300 rows `[x, y, w, h, conf, class]`, NO NMS** — the one-to-one head
  self-dedupes. Labels in `userDefined` metadata `names`. Lands in P3.
- **`DETRSetPredictionDecoder`** — path B, RF-DETR (later, additive; off the
  critical path). sigmoid → flatten top-k → cxcywh→xyxy, no NMS.

**Pin each model's fixed input size + aspect-preserving scale-to-fit** in the
catalog/detector config — never hardcode 640 (RF-DETR variants are 384–880). A
crop/scale that doesn't match how the model was trained silently corrupts boxes.

## What's reused vs. new

**Reused (already exists, M2–M5):** `Detector` / `TunableDetector`,
`DetectorCapabilities`, `DetectorCatalog` / `DetectorCatalogEntry` /
`ActiveDetectorSession` (the M5·P4 selection layer — Core ML models register as
entries exactly like the built-in Vision ones), `DetectorPipeline`, the overlay
(`DetectionLayer` + `VideoGeometry`), the `DetectionInspector`, and the
`ResultStore` / `DetectionCache` path.

**New (this milestone):** `tools/model-export/` Python tooling (P1, done);
`Sources/Iris/Detection/CoreML/` — `CoreMLDetector` + `OutputDecoder` protocol +
`VisionObjectDecoder` (P2) + `YOLOEnd2EndDecoder` (P3); catalog entries + model
loading (bundle + file-picker) (P3); `Captioner` + Foundation Models backend (P4).

## Phases

### P1 — Core ML conversion pipeline  ✅

_Done 2026-05-25. Tooling + runbook + empirically verified YOLO conversion paths
+ converted artifacts._

- **Reusable tooling** under [`tools/model-export/`](../../tools/model-export/):
  `uv`-managed, CLI-first, Python (never a SwiftPM dependency). Blessed default
  is a pinned-ephemeral `uv run --with …` one-liner (`--python 3.12` +
  `numpy<=2.3.5` are load-bearing pins); a persistent `.venv` is the
  repeat-export option. A working `inspect_model.py` (mandatory artifact-verify
  step, coremltools-9 API: `skip_model_load`, `userDefined`,
  `coordinatesOutputFeatureName`).
- **Runbook:** [`tools/model-export/RUNBOOK.md`](../../tools/model-export/RUNBOOK.md)
  — the durable, reusable procedure for converting **any** external detector,
  plus per-model recipes and an "adding the next model" checklist.
- **Verified empirically** (ultralytics 8.4.54, coremltools 9.0, Apple M1 Max):
  - **YOLOv12n → Path A.** `yolo export … nms=True` → Apple
    `NonMaximumSuppression` pipeline, `coordinates` + `confidence` outputs, 80
    COCO labels baked into the NMS stage → Vision auto-decodes, **zero Swift
    decode**. (`nms` defaults to **false** — a bare export is Path B.)
  - **YOLO26n → Path B.** ultralytics **forces `nms=False` on end2end models**
    (warns *"'nms=True' is not available for end2end models"*) → raw `[1,300,6]`
    tensor → a **trivial** Swift decoder: threshold + scale ≤300 rows, **NO NMS**
    (the one-to-one head self-dedupes). Labels in `userDefined` `names`.
  - Converted `yolo11n` / `yolo12n` / `yolo26n` to `.mlpackage`; artifacts +
    layout under [`data/models/`](../../data/models/) (gitignored, regenerable;
    `README.md` tracked).
- **Background:** the exploration

  — [`RECOMMENDATIONS.md`](../../explorations/2026-05-25-coreml-model-conversion/RECOMMENDATIONS.md)

### P3 — `YOLOEnd2EndDecoder` (Path B) + model-swap catalog/loading  📋  ← next

The second decoder through the same seam + the loading story:

- **`YOLOEnd2EndDecoder`** — decodes YOLO26's raw `[1,300,6]`: threshold + scale
  the ≤300 rows, **no NMS** (one-to-one head self-dedupes), labels from
  `userDefined` `names`. Fixture test against the converted `yolo26n`. Proves the
  seam carries a path-B decoder without reshaping `CoreMLDetector` or touching the
  YOLOv12 path.
- **Model-swap catalog/loading** — register Core ML models as `DetectorCatalog`
  entries (factory builds the `CoreMLDetector` with its decoder + `modelIdentifier`);
  support **both** bundled (small FP16, Git LFS, loaded + `prewarm()` at launch)
  and **file-picked** (large/optional/user-supplied, `availability ==
  .modelNotReady` until present) models.

## Open design questions

- 📌 **Runtime-tunable Core ML thresholds — resolved for P2 / re-scoped to P3.**
  Path A (`nms=True`) **bakes the IoU/confidence thresholds at export time**, so
  runtime-tunable thresholds would force Path B (raw tensor + Swift NMS/threshold)
  or a re-export per threshold set — which changes *which decoder* the YOLO catalog
  entries use. **P2 shipped the baked-threshold Path-A `VisionObjectDecoder` and
  conforms `CoreMLDetector` to `Detector` only (no runtime knobs);** the
  genuinely-runtime-tunable variant is re-scoped to P3, decided alongside
  `YOLOEnd2EndDecoder` (the path-B decoder it requires). See
  [`DECISIONS.md`](../DECISIONS.md) (2026-05-26) and
  [`QUESTIONS.md`](../QUESTIONS.md) `[answered 2026-05-26]`.
- 📌 **iOS/macOS 26 auto-decode observation type — resolved.** Confirmed at runtime
  in P2 against the MacOSX26.5 SDK: `CoreMLRequest.Result == [any VisionObservation]`,
  and an NMS object-detection pipeline yields concrete **`RecognizedObjectObservation`**
  (logged via a temporary probe) — `boundingBox: NormalizedRect` (normalized
  lower-left, matches `Detection.boundingBox` — no flip), best-first `labels`,
  `confidence`. The auto-decode *rule* (model output spec, not request type) held.

## Risks

- **Box misalignment from crop/scale mismatch.** The classic Vision↔YOLO trap —
  a center-crop/stretch that doesn't match the model's letterbox training corrupts
  boxes. Mitigation: pin aspect-preserving scale-to-fit per model; the converted
  model declares a fixed-size image input with internal `/255`.
- **Path assumption vs. artifact reality.** Never assume the decode path — the
  RUNBOOK's `inspect_model.py` step makes it a checked fact (`coordinates` +
  `confidence` pipeline ⇒ A; a `multiArrayType` ⇒ B). A doc-only pass already got
  YOLO26's path wrong once; empirical verification corrected it.
- **Foundation Models scope (P4).** Captioning is a stretch; don't let it block the
  Core ML detector close. Sequence it last.
