# M3 — `IrisPlayback`

**Scope.** macOS + iOS. Builds the playback subsystem so a recorded video file
feeds the *same* `AsyncStream<Frame>` pipeline that M1's capture produces,
driving the M2 detection + overlay stack unchanged. macOS is the primary
target — playback is its main entry point per [`../BRIEF.md`](../BRIEF.md) §
Platforms — but the playback source itself is platform-agnostic and runs on
iOS too. First end-to-end exercise of [`PlayerLayerConverter`](../../Sources/Iris/Overlay/PlayerLayerConverter.swift)
(M2 Phase 4 shipped the math; M2 ran the AVF side dry) and of
`DetectionLayer`'s `displayTimeSource` override (M2 Phase 6 baked the closure
in pre-emptively).

**Locked design.** Phases below realize the locked recommendations in:

- [`../../explorations/runtime-pipeline-architecture/RECOMMENDATIONS.md`](../../explorations/runtime-pipeline-architecture/RECOMMENDATIONS.md) — same `Source<Frame>` contract upstream of both `IrisCapture` and `IrisPlayback`
- [`../../explorations/display-pipeline-architecture/RECOMMENDATIONS.md`](../../explorations/display-pipeline-architecture/RECOMMENDATIONS.md) — video-rect-backed converter for playback

## Public surface

**`Sources/Iris/Playback/`:**

- `PlaybackSource` — `Source` conformer wrapping an `AVPlayer` + `AVPlayerItemVideoOutput`. Publishes `AsyncStream<Frame>` keyed by host-clock `CMTime`; `.bufferingNewest(1)` per the [`../DECISIONS.md`](../DECISIONS.md) pipeline contract. Owns `play()` / `pause()` / `step(by: Int)` / `seek(to: CMTime)`. State machine matches `SourceState` (`.idle` / `.running` / `.failed`).
- `PlaybackView: View` — SwiftUI host for the player layer. iOS / macOS parity through a single `NSViewRepresentable` / `UIViewRepresentable` pair returning the underlying `AVPlayerLayer`-backed view. Exposes the `AVPlayerLayer` to callers via an `onPlayerLayerReady:` hook (parallels M2 Phase 7's `CameraPreview.onPreviewLayerReady`) so `PlayerLayerConverter` can be wired to the live layer.
- `Scrubber: View` — SwiftUI thumb-and-track control bound to the player's `currentTime` / `duration`; play / pause / frame-step buttons. Drags scrub `AVPlayer`; frame-step buttons call `PlaybackSource.step(by: ±1)`.

**`Apps/`:**

- `IrisDemo-macOS` — new SwiftUI demo target that mirrors `IrisDemo-iOS`'s shape: pick a video, stack `DetectionLayer` over `PlaybackView`, run `VisionRectanglesDetector` over the frame stream, render boxes via `PlayerLayerConverter`. macOS becomes a self-contained model-eval target.

## Phases

### Phase 1 — `PlaybackSource` core

`Sources/Iris/Playback/PlaybackSource.swift`. `AVPlayer` + `AVPlayerItemVideoOutput`-backed `Source` conformer. `frames` exposes `AsyncStream<Frame>` produced by polling `copyPixelBuffer(forItemTime:itemTimeForDisplay:)` on a `CADisplayLink` (iOS) / `CVDisplayLink` (macOS) tick. Timestamps come from the player item's `itemTime`, *not* host clock — playback `Frame.timestamp` semantics diverge from capture's host-clock timestamps; the [`../../explorations/runtime-pipeline-architecture/RECOMMENDATIONS.md`](../../explorations/runtime-pipeline-architecture/RECOMMENDATIONS.md) contract permits this (timestamps are source-defined) but the divergence wants to be documented on `Frame`. Fixture-based test uses the M2 LFS clip and asserts: frames arrive in monotonic-`CMTime` order, total count is within ±1 of expected (frame rate × duration), and seeking past EOF terminates the stream.

### Phase 2 — Playback controls + `seek` / `step`

`PlaybackSource.play()` / `pause()` / `seek(to:)` / `step(by:)`. `step(by:)` uses `AVPlayerItem.step(byCount:)` for ±1-frame accuracy. Unit tests cover state transitions and that `seek` clamps to `[.zero, duration]`. Frame-step from `.idle` (paused-on-load) needs to yield the frame at the new position *without* the display link being live — verify the polling path supports a synchronous one-shot read.

### Phase 3 — `PlaybackView`

`Sources/Iris/Playback/PlaybackView.swift`. `NSViewRepresentable` / `UIViewRepresentable` pair (single file, `#if os(macOS)` for the NS half) hosting an `AVPlayerLayer`-backed view. `onPlayerLayerReady: @MainActor (AVPlayerLayer) -> Void = { _ in }` defaults to no-op so existing `#Preview` paths don't need it. Internal `PlayerView` (UIKit/AppKit) sized to `videoGravity = .resizeAspect` matching the converter math's assumption. Sanity check: the same `DetectionLayer` works overlay-stacked on `PlaybackView` with no signature change from the M2 iOS demo's `CameraPreview` stack.

### Phase 4 — `Scrubber` + frame-step UI

`Sources/Iris/Playback/Scrubber.swift`. SwiftUI `Slider` bound to `PlaybackSource.currentTime` (an `@Observable` `CMTime` published from the player's `addPeriodicTimeObserver`). Play / pause toggle + `step(by: -1)` / `step(by: 1)` buttons. Drag-to-scrub uses `AVPlayer.seek(to:toleranceBefore: .zero, toleranceAfter: .zero)` for frame-accurate seek (slow, but the UX wants precision over speed). Static `#Preview` with a `MockSource` driving the scrubber's bindings — no AVF in the preview path.

### Phase 5 — `IrisDemo-macOS` demo target

New `Apps/IrisDemo-macOS/` target alongside `IrisDemo-iOS`. `project.yml` gets a second target; share the existing `Local.xcconfig`. Window: file picker → `PlaybackView` with `DetectionLayer` overlay + `Scrubber`. Uses `VisionRectanglesDetector` with the same Phase 7 tuning. End-to-end smoke: open a `.mov`, see detection boxes registered over the played video, scrub and frame-step without overlay drift. First live exercise of `PlayerLayerConverter` (AVF side) and of `DetectionLayer`'s `displayTimeSource: { player.currentTime() }` override.

### Phase 6 — iOS playback parity smoke

Extend `IrisDemo-iOS` with a second tab / mode: playback. Same `PlaybackView` + `Scrubber` + `DetectionLayer` stack. Validates that the playback subsystem is *not* macOS-only — the architecture demands platform-agnostic playback per the locked decision in [`../DECISIONS.md`](../DECISIONS.md) §"macOS parity is a *principle*, not a target". Same fixture clip used in Phase 1's tests.

Land in **Phase 1** alongside the playback orientation work — playback frames need to honor the asset's preferred transform, and the capture-side per-frame orientation question lives in the same conceptual neighborhood.
- **`AVCaptureSession` interruption recovery** (`TODO M1+` in [`CaptureSession.configureSession`](../../Sources/Iris/Capture/CaptureSession.swift)). Land in **Phase 6** alongside the iOS demo extension — the iOS app is where interruptions actually happen (incoming calls, control-center pulldown), and exercising recovery wants both capture and playback present so the user-visible behavior of "swap to playback when capture is interrupted" can be smoke-tested.

## Risks / amendments needed

- **`displayTime` semantic divergence between capture and playback.** Capture frames carry host-clock timestamps (`CMSampleBuffer.presentationTimeStamp`). Playback frames carry asset-time timestamps (`AVPlayerItem.currentTime()`). M2 Phase 6 baked the closure-based `displayTimeSource` in pre-emptively for exactly this — playback callers pass `{ player.currentTime() }` and the per-tick read stays correct. But `ResultStore.lookup(at:)` needs the *same* clock the detector ran on; mixing host-clock detector timestamps with asset-time lookup is a silent staleness bug. Phase 1 must establish the contract: playback's `Frame.timestamp` *is* asset time, and the detector's output inherits that — so `ResultStore` is internally consistent regardless of source. Document this on `Frame` and re-test the staleness thresholds. **Live tracking concern:** even with the per-source contract held, the lookup-vs-detector timestamp match is exact-equality today and reality won't always cooperate (skipped frames, decoder jitter, post-scrub re-emit). Eventual fix is a best-effort temporal match in `ResultStore.lookup` — widen the `stale:` window adaptively, or interpolate between the two nearest `TimestampedDetections` when the lookup time falls between them. Not in M3 scope; tracked as `[open]` in [`../QUESTIONS.md`](../QUESTIONS.md). The `AVAssetReader`-backed prefetch cache (deferred below) is one of the candidate enablers — it lets the detector run ahead of playback so lookup never has to extrapolate.
- **Locked: `AVPlayerItemVideoOutput`-backed `PlaybackSource` for M3.** `AVAssetReader` gives sequential decode at native rate — ideal for *offline* batch processing, painful for *real-time playback with scrubbing*. `AVPlayerItemVideoOutput` lets `AVPlayer` drive playback timing while exposing the same per-frame pixel-buffer pull. `Frame`s from the *same* asset, different access shape. The [`../BRIEF.md`](../BRIEF.md) §IrisPlayback "AVAssetReader-backed" line is a scope note, not a locked architecture decision; M3 supersedes it. **Deferred:** an `AVAssetReader`-backed prefetch/cache source (a `CachedSource` or `OfflineSource`) that decodes frames ahead of playback into a `[CMTime: CVPixelBuffer]` map so the detector can run ahead and `ResultStore` lookups land on exact-match timestamps. Useful both for M5's dataset/eval workflows and as the future enabler for the displayTime-divergence best-effort-match work above. Land when one of those needs it.
- **`AVPlayerLayer` and `AVPlayerItemVideoOutput` are not `Sendable`-clean.** Same escape-hatch doctrine as M1/M2 capture: `@preconcurrency import AVFoundation`, `@unchecked Sendable + NSLock + documented invariant` for the source, per [`../DECISIONS.md`](../DECISIONS.md) §"Strict-concurrency escape hatches".
- **Frame-step ergonomics at video boundaries.** `AVPlayerItem.step(byCount: -1)` at `currentTime == .zero` is a no-op; stepping forward at EOF is also a no-op. The scrubber UI should reflect this without the buttons appearing broken. Phase 4 risk, not Phase 2.
- **No macOS demo target exists yet.** `Apps/` currently holds only `IrisDemo-iOS` + project files. Phase 5 adds a new target; `project.yml` already supports the pattern, so the diff is mechanical. Watch the `Local.xcconfig` sharing — signing for macOS demo can use a different team if needed without forking the file.
- **The locked `PlayerLayerConverter` math assumes `.resizeAspect` videoGravity.** M2 Phase 7's deferred-item list flagged `.resizeAspectFill` vs `.resizeAspect` interaction as untested for `PreviewLayerConverter`. `PlayerLayerConverter` has the same assumption — Phase 3 must set `videoGravity = .resizeAspect` on the demo's `AVPlayerLayer` so the math holds. If a future use case wants fill, the converter math wants a parallel branch (or a different converter); not in M3 scope.

## Exit criteria

- `PlaybackSource` conforms to `Source` and feeds `AsyncStream<Frame>` indistinguishable in shape from `CaptureSession.frames`; the same `DetectorPipeline` and `DetectionLayer` work over it unchanged.
- All public types listed in §Public surface compile under Swift 6 strict concurrency on both iOS and macOS without `@preconcurrency` on the public surface.
- `IrisDemo-macOS` opens a `.mov`, plays it, and overlays detection boxes that stay registered with subjects through play, pause, scrub, and frame-step.
- `IrisDemo-iOS` gains a playback mode that exercises the same code paths on iOS — proving the subsystem is platform-agnostic per the locked macOS-parity principle.
- M2-deferred `SampleBufferRouter` orientation question and `AVCaptureSession` interruption recovery both resolved (or explicitly re-deferred with rationale).
- `PlayerLayerConverter` exercised end-to-end against a live `AVPlayerLayer`; the M2 Phase 4 unit-test math holds in practice.
