# M4 — `IrisTuning`: live filter / threshold controls

**Scope.** First library-side milestone after demo-ergonomics. Make detector knobs adjustable at runtime — confidence thresholds, class filters, NMS, future-detector-specific knobs — without forcing consumers to roll their own settings model from scratch. Land the channel between *settings change* and *what re-runs / re-filters / re-renders*; ship one concrete settings type for `VisionRectanglesDetector` and a thin placeholder UI for the demos. Consumer-side tuning UIs are explicitly out of scope (they own their own themes / layouts) — Iris ships the model and a default surface for smoke-testing.

[`BRIEF.md`](../BRIEF.md) §5 sketches a single concrete `DetectionFilter` (confidence / class allow-block / NMS / min box). That sketch is **a Vision-detector instance** of the settings model, not the model itself. M4 reshapes it into a generic channel that can carry per-detector-family settings, with the Vision shape as the first instance.

## Three-tier change taxonomy *(load-bearing)*

The critical M4 insight is that not every settings change has the same downstream effect. A naïve "every knob change rebuilds the detector" loses two cheap categories. Three tiers:

| Tier               | Example                                                                                   | What invalidates                                                                       | Cost                          |
| ------------------ | ----------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- | ----------------------------- |
| **View-time**      | Box color, line width, label font, "what if?" display style                               | Nothing. Detector unchanged. Cache unchanged. Just re-render.                          | A `View` body invalidation.   |
| **Filter-time**    | Raise confidence threshold above what's already in the cache; remove a class from display | Detector unchanged. Cache unchanged. Filter applied between cache lookup and overlay.  | One pass over a `[Detection]`.|
| **Detector-time**  | Lower confidence below previous minimum; change NMS IoU; add a class the detector skipped | Detector rebuilt (per the 2026-05-20 hot-swap doctrine). Cache entries become stale.   | Full re-inference cost.       |

The settings channel must classify each change into one of these tiers and route accordingly. Misclassifying a filter-time change as detector-time costs latency; misclassifying a detector-time change as filter-time silently produces wrong results (the cache says "no detections at t=12.3" because the *old* threshold filtered them out; the new threshold can't recover them from cache because they were never recorded).

**Confidence threshold is the subtle case.** Raising it above the cached floor is filter-time (we have the higher-confidence detections; just hide the rest). Lowering it below the cached floor is detector-time (the cache doesn't contain the lower-confidence boxes; they were never run through the model with a lower threshold — assuming the detector itself uses the threshold as a cutoff rather than always emitting everything and letting downstream filter). This depends on how each detector implements the knob: some detectors emit *everything* and threshold downstream (filter-time-only), others use the threshold as a model parameter (detector-time on any lowering). The settings channel must let each detector declare its own classification per-knob.

## Settings shape — hybrid: concrete + schema

Per the 2026-05-22 design constraints captured in memory, two needs:

1. **Concrete settings types** for detector families that share anatomy (object detectors broadly: confidence, NMS, class filter, min size). Type-safe binding, easy to use from SwiftUI, clear discoverability.
2. **Schema channel** for detectors whose settings shape diverges sharply (pose estimation: keypoint confidence thresholds per joint; trajectory: smoothing window, max gap; captioning: max-tokens, style). Each detector publishes its setting schema; generic UIs render arbitrary detectors without compile-time knowledge.

Both surfaces coexist. The concrete type is the primary API; the schema export is a parallel read-only view of the same underlying state, for generic UIs and serialization.

```swift
// Sketch — concrete side
public struct VisionRectanglesSettings: DetectorSettings, Sendable {
    public var minimumConfidence: Float
    public var maximumObservations: Int
    public var minimumAspectRatio: Float
    // ... per-knob tier metadata accessible via the schema view
}

// Sketch — schema side
public struct SettingSchema: Sendable {
    public let knobs: [Knob]
    public struct Knob: Sendable {
        public let key: String
        public let label: String
        public let kind: Kind  // .float(range:), .int(range:), .toggle, .multiSelect(options:)
        public let tier: ChangeTier  // .view, .filter, .detector
        // — knobs declare *their own* tier statically; the detector decides if
        //   a transition crosses a tier boundary (e.g. raising vs. lowering confidence).
    }
}

public enum ChangeTier: Sendable { case view, filter, detector }
```

The tier on the `Knob` is the *worst-case* tier — what the channel must assume for an unknown transition. The detector itself can downgrade a specific transition (e.g. "this confidence change is filter-time because the new value is higher than the cached floor") at apply-time.

## Channel: from settings change to effect

```
            ┌──────────────────────────┐
  external  │  TuningModel<Settings>   │  @Observable @MainActor
  SwiftUI ─▶│  (or the schema-keyed    │  — primary public surface
            │   ObservableSettingsStore │
            │   for the schema flavor) │
            └────────────┬─────────────┘
                         │ change event (knob, old, new)
                         ▼
            ┌──────────────────────────┐
            │  Detector.classify(...)  │  detector's tier verdict for *this* transition
            └────────────┬─────────────┘
                         │ .view / .filter / .detector
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         re-render  filter pipeline  rebuild detector
                    (pre-overlay     + invalidate cache
                     filter pass)    entries that used
                                     the old settings
```

The classifier lives on the detector because only the detector knows whether the transition crosses a model parameter or just a display cutoff. The settings model only knows about static (worst-case) tiers from the schema.

## Interaction with `playback-detection-cache`

[`features/playback-detection-cache.md`](./playback-detection-cache.md) Phase 1's [`ResultStore`](../../Sources/Iris/Overlay/ResultStore.swift) caches `TimestampedDetections` keyed by quantized timestamps. M4 introduces detector-time changes that invalidate cache entries. Two options:

1. **Full clear on any detector-time change.** Simplest. Costs every re-traversal of already-played footage after each detector-time tweak.
2. **Tag cache entries with a settings fingerprint.** Each entry records the settings snapshot used to produce it; cache lookup checks fingerprint match. Detector-time changes only invalidate entries whose fingerprint becomes incompatible. More work, much cheaper on iterative tuning.

Decide during execution — start with (1), promote to (2) if the iteration loop is painfully slow in practice. Surface the choice in [`QUESTIONS.md`](../QUESTIONS.md) once Phase 2 lands.

## Public surface

New in `Sources/Iris/Tuning/`:

- `protocol DetectorSettings: Sendable` — marker + `var schema: SettingSchema { get }`.
- `protocol TunableDetector: Detector` — `associatedtype Settings: DetectorSettings`; `var settings: Settings { get }`; `func apply(_ change: SettingChange) -> ApplyResult`. `ApplyResult` carries the tier verdict + optional rebuilt detector.
- `struct VisionRectanglesSettings: DetectorSettings` — concrete shape for the Vision detector.
- `@Observable @MainActor final class TuningModel<Settings: DetectorSettings>` — wraps a settings value, publishes changes, hands them to the detector via the apply protocol. External UI binds to this.
- `struct SettingSchema`, `struct SettingChange`, `enum ChangeTier`, `enum ApplyResult` — supporting types.
- **Built-in UI scaffolding** under `Sources/Iris/Tuning/UI/` — `VisionRectanglesTuningView` (concrete view over the Vision settings), plus a small set of primitives (`TuningSlider`, `TuningToggle`, `ClassFilterChips`) that the concrete view composes and consumers can reuse. Kept narrow. No generic schema renderer in Iris itself — consumers wanting that pattern roll their own from the schema export.

Changes to existing types:

- `VisionRectanglesDetector` gains `Settings = VisionRectanglesSettings` conformance to `TunableDetector` + its `apply(_:)` implementation. Backwards-compatible (default settings reproduce current behavior).
- `DetectorPipeline` learns to consult `TuningModel` and re-route per the tier verdict. Filter-time changes apply a pre-overlay filter; detector-time changes hot-swap and (per the cache decision above) invalidate cache entries.

## Phases

### ✅ Phase 1 — Types + `VisionRectanglesSettings` + `TunableDetector` protocol

Land the protocol shape + the Vision concrete settings type + the schema export. Pure types, no behavior wiring yet. `VisionRectanglesDetector` conforms to `TunableDetector`; `apply(_:)` returns the right tier verdict for confidence raise vs. lower, NMS change, etc. Unit tests against the classifier (every knob × every transition direction → expected tier).

### ✅ Phase 2 — `TuningModel<Settings>` + pipeline wiring

`@Observable @MainActor TuningModel<Settings>` publishes settings changes. `DetectorPipeline` accepts an optional `TuningModel` and routes change events: view-tier ignored (overlay re-renders on its own through the existing `ResultStore` publish); filter-tier installs a pre-overlay filter pass; detector-tier rebuilds the detector via the existing hot-swap doctrine and clears the cache (option 1 above — simplest first). Tests: a recording-detector double observes that detector-tier changes rebuild and view-tier changes do not.

### ✅ Phase 3 — Built-in UI: `VisionRectanglesTuningView` + primitives

Ship the concrete view + the three primitives (`TuningSlider`, `TuningToggle`, `ClassFilterChips`) it composes. Wire both demos (iOS + macOS) to expose the view — iOS in a sheet from a toolbar button on the Playback tab; macOS in a secondary sidebar or `Inspector` panel. Visual previews under `#Preview` for each primitive at known settings values. No tests for the views themselves; the model logic is already covered in Phase 2.

### 🚫 Phase 4 — Filter-time pipeline pass + cache-fingerprint upgrade *(conditional)*

If Phase 3 smoke shows iterative tuning is painfully slow on already-cached footage, promote the cache from "full clear on detector-tier" to "fingerprint per entry". Otherwise close M4 with Phase 3 and surface the fingerprint work as a deferred item.

## Risks

- **Tier misclassification silently produces wrong overlays.** A confidence change classified as filter-time when it should be detector-time leaves the cache with stale "no detection" answers that don't recover. Mitigation: per-detector `classify(_:)` tests at Phase 1 exhaustively cover every knob × direction; the schema's static tier defaults to the worst case (`.detector`) so the unsafe failure mode is "rebuilds too often", not "stale answers".
- **`TuningModel<Settings>` generic erasure.** External UIs that want to render multiple detector types generically need an existential or a schema-driven path. The dual surface (concrete + schema) covers both; the cost is two API surfaces to keep aligned. Mitigation: schema is derived from the concrete type rather than maintained separately.
- **Cache invalidation correctness vs. cost.** Option (1) full-clear is correct but slow; option (2) fingerprinted is fast but easy to get subtly wrong (which knobs participate in the fingerprint?). Mitigation: ship (1) first, gather usage data, decide (2) on evidence not speculation.
- **Schema-published-by-detector duplicates concrete knowledge.** `VisionRectanglesSettings` defines its knobs in code; the schema needs to re-state them. Drift risk. Mitigation: derive the schema from the concrete type via a `@SettingsSchema` macro or a static method on the settings type that walks its own properties (Swift 6 has the macro tools for this). Phase 1 picks a derivation strategy.
- **Built-in UI scope creep.** "One thin view per known detector + primitives" can grow into "Iris ships a full tuning design system". Mitigation: Phase 3's exit criterion is *just* the Vision view + three primitives. Anything else is a separate feature.

## Open design questions *(land in [`QUESTIONS.md`](../QUESTIONS.md) at Phase 1)*

- **Cache fingerprinting strategy.** Which knobs participate? Confidence floor obviously; NMS yes; "what if?" display tier no. Resolve when Phase 4 opens.
- **Settings persistence.** Does `TuningModel` persist to `UserDefaults` like the demo's `RecentVideos`, or is persistence a consumer concern? Lean consumer-concern (matches the M3 doctrine of UI-shaped state outside the package), but the demos will want it.
- **Multi-detector pipelines.** When a `DetectorPipeline` runs multiple detectors, does each get its own `TuningModel`, or is there a composite? Phase 2 ships with one-detector-one-model; composite is a deferred item.
- **"What if?" mode from BRIEF §5.** Show detections that *would* pass at a lower threshold in a different style. This is a view-tier overlay on a detector-tier change (you need the lower-threshold detections cached). Cleanest path: a dedicated `previewSettings` channel that runs alongside live settings, materializing the cache for the preview values; the overlay renders both layers. Deferred to a follow-up feature; M4 ships without it.

- `Sources/Iris/Tuning/` exists with `DetectorSettings`, `TunableDetector`, `TuningModel`, `SettingSchema`, `SettingChange`, `ChangeTier`, `ApplyResult`, and `VisionRectanglesSettings`.
- `VisionRectanglesDetector` conforms to `TunableDetector`; backward-compatible (existing tests pass with default settings).
- `DetectorPipeline` routes changes via the tier verdict; detector-tier rebuilds + cache-clears; view-tier is a no-op at the pipeline; filter-tier applies a pre-overlay pass.
- `VisionRectanglesTuningView` + `TuningSlider` / `TuningToggle` / `ClassFilterChips` ship under `Sources/Iris/Tuning/UI/`.
- Both demos surface the tuning view (iOS sheet; macOS sidebar/inspector). Live knob changes visibly affect the overlay on Vision detections.
- Swift 6 strict concurrency clean. `swift build`, `swift test`, both `xcodebuild` schemes green.
- Test additions cover the classifier (every Vision knob × direction → expected tier), the `TuningModel` change publication, and the pipeline routing per tier.
- Open questions land in [`QUESTIONS.md`](../QUESTIONS.md); the cache-fingerprint follow-up lands as a deferred item if Phase 4 doesn't ship.
