## Open

- [open 2026-05-23] **Cache fingerprinting strategy for `ResultStore` under M4.** Detector-tier settings changes invalidate cache entries produced under the old knobs. Option (1) full-clear is correct but throws away every prior visit; option (2) tags each entry with a settings fingerprint and only invalidates entries whose fingerprint becomes incompatible. M4 ships (1) at Phase 2 and promotes to (2) only if iterative tuning on already-cached footage is painfully slow (Phase 4 trigger). Which knobs participate in the fingerprint when (2) lands? Confidence floor and the aspect / size / NMS-style knobs obviously; `label` and view-tier knobs do not. Resolve when Phase 4 opens (or surface as deferred if Phase 4 doesn't ship). See [`features/M4.md`](./features/M4.md) §"Interaction with `playback-detection-cache`" and §"Open design questions".

- [open 2026-05-23] **Settings persistence — `TuningModel` vs. consumer responsibility.** Does `TuningModel<Settings>` persist its current values to `UserDefaults` (mirroring M3's `RecentVideos`), or is persistence purely a consumer concern? Leaning consumer-concern — matches the M3 doctrine of UI-shaped state living outside the package — but the demos will want a stored-tuning experience for smoke testing iterative knob changes across launches. Resolve at M4 Phase 3 when the demos surface the tuning view. See [`features/M4.md`](./features/M4.md) §"Open design questions".

- [open 2026-05-23] **Multi-detector pipelines under `TuningModel`.** When a `DetectorPipeline` runs more than one detector, does each detector get its own `TuningModel<Settings>`, or is there a composite settings model spanning them? M4 Phase 2 ships with the one-detector-one-model shape; the composite design is deferred. Resolve when a real multi-detector pipeline lands (post-M4). See [`features/M4.md`](./features/M4.md) §"Open design questions".

- [open 2026-05-23] **"What if?" mode from `BRIEF.md` §5.** Show detections that *would* pass at a lower threshold rendered in a distinct style — a view-tier overlay over a detector-tier change (you need the lower-threshold detections cached to display them). Cleanest sketch: a dedicated `previewSettings` channel running alongside live settings, materializing the cache for the preview values; the overlay renders both layers. Deferred to a follow-up feature; M4 ships without it. See [`features/M4.md`](./features/M4.md) §"Open design questions" and [`BRIEF.md`](./BRIEF.md) §5.

## Answered

- [answered 2026-05-22] **`displayTime` semantic divergence between capture (host clock) and playback (asset time), and the exact-equality assumption in `ResultStore.lookup`.** Resolved by [`features/playback-detection-cache.md`](./features/playback-detection-cache.md) — `ResultStore` is now a quantized-timestamp-keyed dictionary cache with nearest-neighbor `lookup(at:)` within `min(2 × quantization, stale:)`. The per-source `Frame.timestamp` contract still differs (capture = host clock, playback = asset time), but lookup no longer assumes exact equality, so the divergence stops mattering at the millisecond scale that tripped up real playback. The originally-suspected exact-equality assumption turned out to be a *minor* factor; the load-bearing failure was actually `ResultStore`'s 30-frame ring buffer evicting most playback history, and the lack of detection-result reuse across re-visits. Both fixed in the same work. See [`DECISIONS.md`](./DECISIONS.md) §"Best-effort temporal match in `ResultStore.lookup`".
- [answered 2026-05-21] **`Detector` stateful-conformer shape.** Protocol stays `Sendable`-only; stateful conformers wrap state in an internal `actor` by their own choice. Already covered by the two 2026-05-20 decisions in [`DECISIONS.md`](./DECISIONS.md): "Hot-swap by replacing the instance" + "`@CaptureActor` in `IrisCapture`'s public API". Forcing actor-ness at the protocol level (`AnyObject` etc.) buys no real type-safety since the type system can't tell "actor" from "Sendable class with NSLock" — both are legitimate per the strict-concurrency escape-hatch decision. Ergonomic shape of a stateful actor conformer (call-site overhead crossing the actor boundary) gets validated when the first stateful conformer ships — not in M2's locked scope.
- [answered 2026-05-21] **Package layout fork.** Stay with the current single-target shape; revisit only when adapter packages actually need to grow separately. Covered by the existing 2026-05-20 decision — splitting later is a non-breaking change. See [`DECISIONS.md`](./DECISIONS.md) §"Single SwiftPM target, folder-organized internally".
- [answered 2026-05-20] Async model: `AsyncStream<Frame>` exposed via an `AsyncSequence` protocol, `.bufferingNewest(1)` from day one. See [`DECISIONS.md`](./DECISIONS.md).
- [answered 2026-05-20] Concurrency boundaries: `@CaptureActor` in `IrisCapture` public API; not extended to `Detector`. See [`DECISIONS.md`](./DECISIONS.md).
- [answered 2026-05-20] Hot-swapping Core ML: swap the instance, never mutate in place. `Detector: Sendable`; stateless conformers `struct`, stateful conformers `actor`. See [`DECISIONS.md`](./DECISIONS.md).
- [answered 2026-05-20] macOS overlay parity: SwiftUI `Canvas` + centralized Y-flip + `NormalizedGeometryConverting` protocol with per-source backends. See [`DECISIONS.md`](./DECISIONS.md).
- [answered 2026-05-20] Foundation Models scope: two protocols (`Detector` + `Captioner`); VLM backends conform to both. See [`DECISIONS.md`](./DECISIONS.md).
